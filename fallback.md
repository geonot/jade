## Section 1: Features With Multiple Syntactical Expressions

### 1a. `xor` keyword vs `^` symbol — ⚠️ DIFFERENT PRECEDENCE BUG

- `xor` is parsed at **logical** precedence (between `or` and `and`) in `parse_xor` — [expr.rs](src/parser/expr.rs#L148)
- `^` is parsed at **bitwise** precedence (between `|` and `&`) in `parse_bitxor` — [expr.rs](src/parser/expr.rs#L217)
- Both map to `BinOp::BitXor`
- **Impact**: `a xor b + 1` parses differently from `a ^ b + 1`
- **Recommendation**: Remove `^` as xor (keep `xor` keyword), or unify precedence

### 1b. `!` vs `!!` error return — IDENTICAL BEHAVIOR

- `! expr` → [stmt.rs](src/parser/stmt.rs#L223-L227) → `Stmt::ErrReturn`
- `!! expr` → [stmt.rs](src/parser/stmt.rs#L229-L233) → `Stmt::ErrReturn`
- Both produce the exact same AST node
- **Recommendation**: Pick one and remove the other

### 1c. `array[...]` vs `[...]` — DUPLICATED PARSER CODE

- `array[1, 2, 3]` — [expr.rs](src/parser/expr.rs#L429-L465) under `Token::Array`
- `[1, 2, 3]` — [expr.rs](src/parser/expr.rs#L466-L495) under `Token::LBracket`
- ~30 lines of duplicated parsing logic, both produce `Expr::Array`
- **Recommendation**: Remove `array[...]` syntax (it's verbose) or factor out shared code

### 1d. `%` symbol vs `mod` keyword — BOTH ACTIVE

- `%` → `Token::Percent` via symbol lexing
- `mod` → `Token::Percent` via keyword map — [lexer.rs](src/lexer.rs#L297)
- Both produce identical results
- **Note**: `%` is also used as address-of in type context, creating ambiguity
- **Recommendation**: Decide if `%` should be modulo, address-of, or removed

### 1e. `eq`/`equals` — TWO SPELLINGS, ONE TOKEN

- Both → `Token::Equals` in the keyword map
- **Recommendation**: Keep both (short + long form is intentional)

### 1f. `neq` vs `not equals` — TWO-TOKEN PARSE

- `neq` → single token `Token::Neq`
- `not equals` → two-token parse in [expr.rs](src/parser/expr.rs#L165-L168)
- Both produce `BinOp::Ne`
- **Recommendation**: Keep both (intentional)

### 1g. `from` vs `in` in for-loops — SEMANTIC DISTINCTION

- `for i from 0 to 10` — range iteration
- `for i in collection` — collection iteration
- Same parser path, both accepted
- **Recommendation**: Keep — clear semantic distinction

### 1h. `unless`/`until` — INTENTIONAL SUGAR

- `unless cond` → desugars to `if not cond`
- `until cond` → desugars to `while not cond`
- **Recommendation**: Keep — part of Jade's English readability design

### 1i. `do...end` vs indentation — LAMBDA BLOCKS

- Both ways to write multi-statement lambda bodies
- `do...end` enables inline lambdas without indentation
- **Recommendation**: Keep — serves legitimate use case

### 1j. Comparison aliases: `nlt`/`ngt`/`ngte`/`nlte`

- English negated forms: "not less than" = `>=`, etc.
- **Recommendation**: Keep — English readability

### 4b. `Token::StarStar` phantom in parser

- `Token::StarStar` is checked in `parse_exp` at [expr.rs](src/parser/expr.rs#L225)
- But `**` is no longer lexed as `StarStar` — only `pow` keyword maps to it
- The parser check works correctly (since `pow` → `StarStar`), but the token name is misleading
- **Consider**: Rename `Token::StarStar` to `Token::Pow` for clarity

### 4c. `Token::Set` — dual use

- `Token::Set` is a keyword (`set`) used for:
  1. Store `set` operations (active, working)
  2. `Type::Set` construction (dead, see 2g above)
- If `Type::Set` is removed, `Token::Set` should remain (for stores)
