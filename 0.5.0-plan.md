## 1. P0 — Package Manager

### 1.1 Design Philosophy

No centralized registry. No npm-style registry server to operate, defend, or fund. Packages are git repos. Resolution is deterministic. The lockfile is the law.

Why git-backed:
- Every package already has a URL, a hash, a DAG of history — git provides all three for free
- Offline clones work. Vendoring is `cp -r`. Auditing is `git log`
- GitHub/GitLab/Codeberg/self-hosted all work day one — zero vendor lock-in
- Cargo, Go modules, and Zig all converge on this model because it's correct

### 1.2 Manifest Format: `jade.pkg`

Plain text, no TOML/YAML/JSON parser needed. The compiler already has an excellent lexer — reuse it.

```
package counter
version 0.1.0
author Rome

require io https://github.com/jade-lang/std-io v0.1.0
require math https://github.com/jade-lang/std-math v0.1.0
require http https://github.com/user/jade-http v2.3.1
```

**Grammar:**

```
pkg_file     = directive*
directive    = pkg_decl | version_decl | author_decl | require_decl
pkg_decl     = "package" IDENT NEWLINE
version_decl = "version" VERSION NEWLINE
author_decl  = "author" STRING NEWLINE
require_decl = "require" IDENT URL VERSION NEWLINE
VERSION      = DIGIT+ "." DIGIT+ "." DIGIT+
URL          = [a-zA-Z0-9_:/.@-]+
```

**Why not TOML:** Adding a TOML parser is ~2K LOC of dependency or ~500 LOC hand-rolled. The Jade lexer already handles identifiers, strings, and newlines. A 50-line parser on top of the existing token stream handles everything. Zero new dependencies. This is the Jade way.

### 1.3 Lockfile: `jade.lock`

Content-addressable. Every dependency pinned to an exact git commit hash. Deterministic, auditable, diffable.

```
# jade.lock — auto-generated, do not edit
io https://github.com/jade-lang/std-io v0.1.0 a1b2c3d4e5f6
math https://github.com/jade-lang/std-math v0.1.0 b2c3d4e5f6a7
http https://github.com/user/jade-http v2.3.1 c3d4e5f6a7b8
  http.tls https://github.com/jade-lang/tls v0.2.0 d4e5f6a7b8c9
```

Transitive dependencies indented under their parent. Flat when no conflicts. The lockfile is the complete truth — if it exists, `jade.lock` is read and no resolution happens. Only `jadec pkg update` re-resolves.

### 1.4 Directory Layout

```
~/.jade/
  cache/
    github.com/
      jade-lang/
        std-io/
          v0.1.0/           ← git clone at pinned commit
            jade.pkg
            src/
              io.jade
              file.jade
              ...
```

Packages cloned once into `~/.jade/cache/`. Symlinked or path-resolved at compile time. No per-project `node_modules` bloat. Shared cache across all projects, keyed by URL + commit hash.

### 1.5 Module Resolution Changes

Current `resolve_modules` (main.rs:59-104) resolves `use a.b.c` as:
1. `{base_dir}/a/b/c.jade`
2. `{base_dir}/std/c.jade` (fallback)

New resolution order:
1. **Local:** `{base_dir}/{a/b/c}.jade` — same-project modules
2. **Package:** `{cache}/{a}/src/{b/c}.jade` — if `a` matches a `require` name in `jade.pkg`
3. **Std fallback:** `{base_dir}/std/{c}.jade` — backward compat (deprecated)
4. **Bundled std:** `{jade_install}/std/{c}.jade` — shipped with compiler

The `use` syntax already supports dotted paths (`use io.file` → `["io", "file"]`). The first segment is checked against `require` names. If it matches a package, route to cache. Otherwise, local file resolution.

### 1.6 Implementation Plan

#### 1.6.1 Manifest Parser — `src/pkg.rs` (new, ~150 LOC)

```rust
pub struct Package {
    pub name: String,
    pub version: SemVer,
    pub author: Option<String>,
    pub requires: Vec<Dependency>,
}

pub struct Dependency {
    pub name: String,
    pub url: String,
    pub version: SemVer,
}

pub struct SemVer {
    pub major: u32,
    pub minor: u32,
    pub patch: u32,
}

impl Package {
    pub fn parse(input: &str) -> Result<Self, String> { ... }
    pub fn from_file(path: &Path) -> Result<Self, String> { ... }
}
```

Hand-rolled line-by-line parser. No regex, no external crate. Split on whitespace, match first token:
- `"package"` → take ident
- `"version"` → parse `X.Y.Z`
- `"author"` → take rest of line
- `"require"` → take ident, URL, version

Fail fast with line numbers on malformed input.

#### 1.6.2 Lockfile Parser/Writer — `src/lock.rs` (new, ~120 LOC)

```rust
pub struct LockEntry {
    pub name: String,
    pub url: String,
    pub version: SemVer,
    pub commit: String,      // 40-char hex SHA
    pub deps: Vec<LockEntry>, // transitive
}

pub struct Lockfile {
    pub entries: Vec<LockEntry>,
}

impl Lockfile {
    pub fn parse(input: &str) -> Result<Self, String> { ... }
    pub fn write(&self) -> String { ... }
    pub fn from_file(path: &Path) -> Result<Self, String> { ... }
}
```

Parse: indentation-based (0 = direct dep, 2 = transitive). Each line: `name url version commit`.  
Write: deterministic sort by name, then by depth. Byte-reproducible output.

#### 1.6.3 Cache Manager — `src/cache.rs` (new, ~200 LOC)

```rust
pub struct Cache {
    root: PathBuf,  // ~/.jade/cache
}

impl Cache {
    pub fn new() -> Self { ... }
    pub fn package_path(&self, dep: &Dependency) -> PathBuf { ... }
    pub fn is_cached(&self, dep: &Dependency, commit: &str) -> bool { ... }
    pub fn fetch(&self, dep: &Dependency) -> Result<String, String> { ... }
    pub fn checkout(&self, dep: &Dependency, commit: &str) -> Result<PathBuf, String> { ... }
}
```

**`fetch` implementation:**
1. Compute cache dir: `~/.jade/cache/{url_to_path}/{version}/`
2. If dir exists → return existing commit hash
3. `git clone --depth=1 --branch v{version} {url} {cache_dir}` via `std::process::Command`
4. Read `git rev-parse HEAD`, store commit hash
5. Return commit hash

**`checkout` implementation:**
1. Check if `{cache_dir}` exists at correct commit
2. If not: `git fetch origin {commit}` + `git checkout {commit}`
3. Return path to package root

**Why shallow clone:** `--depth=1` fetches ~100KB instead of full history. For pinned versions this is always sufficient. If a user needs to inspect history, they clone the repo themselves.

**Why shell out to git:** Git's wire protocol is ~10K LOC to implement natively. Shelling out to `git` is 5 lines, works on every system where Jade will be used, and matches what Go, Zig, and early Cargo do. When/if this becomes a bottleneck, swap in `libgit2` bindings.

#### 1.6.4 Resolution Algorithm — in `src/cache.rs` (~80 LOC)

```rust
pub fn resolve(pkg: &Package, lock: Option<&Lockfile>) -> Result<Lockfile, String> { ... }
```

**Algorithm:**
1. If lockfile exists and all `require` entries match → return lockfile (zero work)
2. For each `require` in `jade.pkg`:
   a. If present in lockfile and version matches → reuse pinned commit
   b. Otherwise → `cache.fetch(dep)` to get commit hash
   c. Read the package's own `jade.pkg` → recursively resolve its requires
3. Detect cycles: maintain `HashSet<String>` of in-progress resolutions → error on cycle
4. Flatten transitive deps. On version conflict (same package, different versions) → **error with both paths shown**. No silent version resolution. The developer decides.

**Why no SAT solver:** Jade uses exact versions, not semver ranges. `require foo v1.2.3` means exactly `v1.2.3`, not `^1.2.3`. This makes resolution trivial — linear scan, no backtracking. Version ranges add combinatorial complexity (the `pubgrub` algorithm is 3K LOC) for dubious benefit. If you want a different version, change the number. Explicit is better than magic.

#### 1.6.5 Module Resolution Integration — `src/main.rs` changes (~60 LOC)

Modify `resolve_modules` to accept a `resolved: &HashMap<String, PathBuf>` mapping package names to their cache paths:

```rust
fn resolve_modules(
    prog: &mut Program,
    base_dir: &Path,
    loaded: &mut HashSet<String>,
    packages: &HashMap<String, PathBuf>,  // NEW: name → cache path

    // ... existing use extraction ...

            c

        // ... rest of loading logic unchanged ...

```

In `main()`, before `resolve_modules`:

```rust
// Load package manifest if present
let pkg_file = base_dir.join("jade.pkg");
let packages = if pkg_file.exists() {
    let pkg = Package::from_file(&pkg_file).unwrap_or_else(|e| die(&format!("jade.pkg: {e}")));
    let lock_file = base_dir.join("jade.lock");
    let lock = if lock_file.exists() {
        Some(Lockfile::from_file(&lock_file).unwrap_or_else(|e| die(&format!("jade.lock: {e}"))))
    } else {
        None
    };
    let cache = Cache::new();
    let resolved = cache.resolve(&pkg, lock.as_ref())
        .unwrap_or_else(|e| die(&format!("resolve: {e}")));
    // Write lockfile if it changed
    let lock_content = resolved.write();
    fs::write(&lock_file, &lock_content).unwrap_or_else(|e| die(&format!("write lock: {e}")));
    // Build name → path map
    resolved.entries.iter()
        .map(|e| (e.name.clone(), cache.package_path_from_entry(e)))
        .collect::<HashMap<_, _>>()
} else {
    HashMap::new()
};
resolve_modules(&mut prog, base_dir, &mut loaded, &packages);
```

## 2. P0 — Standard Library

### 2.1 Design Philosophy

The standard library is Jade source code. Not C wrappers. Not runtime magic. Pure `.jade` files that use `extern` for syscalls and libc functions, then build abstractions in Jade itself. When you `use std.io`, you get inlined, monomorphized, zero-overhead code — the same as if you'd written it by hand.

**Guiding principles:**
- **Thin over libc:** Don't reimplement `printf`. Wrap libc functions with Jade types, let LLVM inline the wrapper.
- **Value types first:** Return structs, not error codes. `file.read()` returns `Result of String, IOError`, not `-1`.
- **No hidden allocation:** Every allocation visible. `read_file(path)` mallocs a buffer — the function signature tells you (`-> String`).
- **Platform-specific behind extern:** Platform differences live in `extern` declarations. Jade code above them is portable.

### 2.2 Module Layout

```
std/
  io.jade          — File I/O, stdin/stdout/stderr
  os.jade          — Environment, args, exit, getcwd, signals
  math.jade        — Transcendental functions, constants, abs/min/max
  fmt.jade         — String formatting, number-to-string conversion
  str.jade         — Extended string utilities beyond builtins
  net.jade         — TCP/UDP sockets (Phase 2)
  time.jade        — Clock, duration, sleep
  sort.jade        — Sorting algorithms for arrays/Vec
  hash.jade        — Hash functions (FNV, xxHash)
  path.jade        — Path manipulation
  rand.jade        — PRNG (xoshiro256**)
```

Shipped with the compiler binary. At install time, `std/` lives alongside `jadec`. The compiler resolves `use std.io` to `{install_dir}/std/io.jade` as a final fallback after local and package resolution.

#### 2.3.3 Core Functions

```jade
*read_file(path: String) -> String
    # Open, seek to end for size, rewind, bulk read, close
    # Single malloc, single fread — zero copies
    fp is fopen(&path, "r")
    if fp as i64 equals 0
        return ""
    fseek(fp, 0, 2)          # SEEK_END
    size is ftell(fp)
    fseek(fp, 0, 0)          # SEEK_SET
    buf is malloc(size + 1)
    fread(buf, 1, size, fp)
    fclose(fp)
    # Build String from buf — takes ownership of malloc'd buffer
    __string_from_raw(buf, size)

*write_file(path: String, content: String)
    fp is fopen(&path, "w")
    if fp as i64 equals 0
        return false
    fwrite(&content, 1, content.length, fp)
    fclose(fp)
    true

*append_file(path: String, content: String)
    fp is fopen(&path, "a")
    if fp as i64 equals 0
        return false
    fwrite(&content, 1, content.length, fp)
    fclose(fp)
    true

*open(path: String, mode: String)
    fp is fopen(&path, &mode)
    File(fp, path, mode)

*File.read self, size: i64
    buf is malloc(size)
    n is fread(buf, 1, size, self.handle)
    __string_from_raw(buf, n)

*File.write self, data: String
    fwrite(&data, 1, data.length, self.handle)

*File.close self
    fclose(self.handle)

*File.eof self
    feof(self.handle) isnt 0
```

**Implementation note:** `__string_from_raw(ptr, len)` is a new compiler builtin that constructs a `String` struct from a raw pointer + length without copying. The pointer becomes the String's data pointer. This avoids a double malloc+memcpy for every file read. Add as `BuiltinFn::StringFromRaw` in hir.rs, codegen in builtins.rs as a `build_string(ptr, len, len)` call.

#### 2.3.4 Stdin/Stdout/Stderr

```jade
extern *__jade_stdin() -> %i8     # returns stdin FILE*
extern *__jade_stdout() -> %i8    # returns stdout FILE*
extern *__jade_stderr() -> %i8    # returns stderr FILE*

*print(s: String)
    fwrite(&s, 1, s.length, __jade_stdout())

*println(s: String)
    print(s)
    fwrite("\n", 1, 1, __jade_stdout())

*eprint(s: String)
    fwrite(&s, 1, s.length, __jade_stderr())

*read_line()
    buf is malloc(4096)
    result is fgets(buf, 4096, __jade_stdin())
    if result as i64 equals 0
        free(buf)
        return ""
    len is strlen(buf)
    # Strip trailing newline
    if len > 0 and @(buf + len - 1) as i64 equals 10
        len is len - 1
    __string_from_raw(buf, len)
```

**`__jade_stdin/stdout/stderr` implementation:** These are trivial C stubs (3 lines each) that return the global `stdin`/`stdout`/`stderr` pointers. Ship as a 20-line `runtime.c` compiled to `libjade_rt.a` linked automatically. Alternatively, emit them as LLVM global externals pointing to the libc symbols — no C stub needed, just `@stdin = external global ptr` in the LLVM module.

The cleanest approach: declare `stdin`/`stdout`/`stderr` as external globals in codegen when `std.io` is imported:

```rust
// In codegen, when io module is loaded:
let stdin_global = self.module.add_global(ptr_type, None, "stdin");
stdin_global.set_linkage(Linkage::External);
```

Then `__jade_stdin()` just loads from the global. Zero C code needed.

### 2.6 `std.fmt` — String Formatting (~180 LOC)

Jade already has string interpolation (`'hello {name}'`). `std.fmt` provides explicit formatting control for cases where interpolation isn't sufficient.

```jade
*int_to_string(n: i64) -> String
    # Already exists as to_string() builtin
    to_string(n)

*float_to_string(x: f64, decimals: i64) -> String
    # snprintf-based formatting with precision control
    __fmt_float(x, decimals)

*pad_left(s: String, width: i64, ch: i64) -> String
    if s.length >= width
        return s
    padding is width - s.length
    result is ""
    for i from 0 to padding
        result is result + char(ch)
    result + s

*pad_right(s: String, width: i64, ch: i64) -> String
    if s.length >= width
        return s
    padding is width - s.length
    result is s
    for i from 0 to padding
        result is result + char(ch)
    result

*hex(n: i64) -> String
    __fmt_hex(n)

*oct(n: i64) -> String
    __fmt_oct(n)

*bin(n: i64) -> String
    __fmt_bin(n)

*join(items: Vec of String, sep: String) -> String
    if items.length equals 0
        return ""
    result is items.get(0)
    for i from 1 to items.length
        result is result + sep + items.get(i)
    result

*repeat(s: String, n: i64) -> String
    result is ""
    for i from 0 to n
        result is result + s
    result
```

**`__fmt_float`, `__fmt_hex`, `__fmt_oct`, `__fmt_bin` implementation:** New builtins that call `snprintf` with appropriate format strings (`"%.Nf"`, `"%lx"`, `"%lo"`, converting to binary via bit ops). ~30 lines each in builtins.rs. The `snprintf` function is already declared in codegen (`ensure_snprintf`).

### 2.8 `std.sort` — Sorting (~100 LOC)

```jade
# Introsort: quicksort + heapsort fallback + insertion sort for small partitions
# Same algorithm as Rust's sort_unstable and C++'s std::sort

*sort(arr: Vec of i64)
    __introsort(arr, 0, arr.length - 1, 2 * __floor_log2(arr.length))

*__introsort(arr: Vec of i64, lo: i64, hi: i64, depth_limit: i64)
    if hi - lo < 16
        __insertion_sort(arr, lo, hi)
        return
    if depth_limit equals 0
        __heapsort(arr, lo, hi)
        return
    pivot is __partition(arr, lo, hi)
    __introsort(arr, lo, pivot - 1, depth_limit - 1)
    __introsort(arr, pivot + 1, hi, depth_limit - 1)

*__partition(arr: Vec of i64, lo: i64, hi: i64) -> i64
    # Median-of-three pivot selection
    mid is (lo + hi) / 2
    if arr.get(mid) < arr.get(lo)
        __swap(arr, lo, mid)
    if arr.get(hi) < arr.get(lo)
        __swap(arr, lo, hi)
    if arr.get(mid) < arr.get(hi)
        __swap(arr, mid, hi)
    pivot is arr.get(hi)
    i is lo - 1
    for j from lo to hi
        if arr.get(j) <= pivot
            i is i + 1
            __swap(arr, i, j)
    __swap(arr, i + 1, hi)
    i + 1

*__insertion_sort(arr: Vec of i64, lo: i64, hi: i64)
    for i from lo + 1 to hi + 1
        key is arr.get(i)
        j is i - 1
        while j >= lo and arr.get(j) > key
            arr.set(j + 1, arr.get(j))
            j is j - 1
        arr.set(j + 1, key)

*__swap(arr: Vec of i64, a: i64, b: i64)
    tmp is arr.get(a)
    arr.set(a, arr.get(b))
    arr.set(b, tmp)
```

**Why introsort:** O(n log n) worst case (no quicksort O(n²) degenerate behavior), O(1) extra space (in-place), small constant factor, cache-friendly sequential access pattern. This is what every serious standard library uses (libstdc++, Rust, .NET).

**Generic sort:** Requires trait bounds (`T: Ord`), planned in Phase 0.3.0. For now, provide `sort_i64`, `sort_f64`, `sort_strings` as separate functions. When trait bounds land, unify into generic `sort of T: Ord`.

### 2.9 `std.rand` — Random Numbers (~80 LOC)

```jade
# xoshiro256** — fast, high-quality, 256-bit state PRNG
# Used by Rust, Julia, Lua. Period: 2^256 - 1.

type RNG
    s0: u64
    s1: u64
    s2: u64
    s3: u64

*seed(n: u64) -> RNG
    # SplitMix64 expansion from single seed to 4×64-bit state
    z is n
    z is (z ^ (z >> 30)) * 0xbf58476d1ce4e5b9 as u64
    s0 is z
    z is (z ^ (z >> 27)) * 0x94d049bb133111eb as u64
    s1 is z
    z is (z ^ (z >> 31))
    s2 is z
    z is (z * 0xbf58476d1ce4e5b9 as u64) ^ (z >> 30)
    s3 is z
    RNG(s0, s1, s2, s3)

*RNG.next_u64 self -> u64
    result is rotate_left(self.s1 * 5, 7) * 9
    t is self.s1 << 17
    self.s2 is self.s2 ^ self.s0
    self.s3 is self.s3 ^ self.s1
    self.s1 is self.s1 ^ self.s2
    self.s0 is self.s0 ^ self.s3
    self.s2 is self.s2 ^ t
    self.s3 is rotate_left(self.s3, 45)
    result

*RNG.next_i64 self -> i64
    self.next_u64() as i64

*RNG.next_f64 self -> f64
    # [0.0, 1.0) — standard uniform
    (self.next_u64() >> 11) as f64 * 5.421010862427522e-20

*RNG.range self, lo: i64, hi: i64 -> i64
    # Unbiased bounded random via rejection sampling
    span is (hi - lo) as u64
    threshold is (0 - span) % span  # = (2^64 - span) % span
    r is self.next_u64()
    while r < threshold
        r is self.next_u64()
    lo + (r % span) as i64
```

**Why xoshiro256\*\*:** Fastest non-trivial PRNG (< 1ns per call), passes all BigCrush tests, 256-bit state fits in 4 registers. Used by Rust (`rand` crate default), Julia (default), Lua 5.4. No external dependencies. Pure arithmetic — compiles to ~15 instructions, LLVM will inline completely.

## 3. P1 — LSP Server

### 3.1 Architecture

The LSP server is a **separate binary** (`jadec-lsp`) that reuses the compiler library crate. It links against the same `jadec` crate (`use jadec::{lexer, parser, typer, ...}`), runs the pipeline up to HIR (no codegen, no LLVM), and serves diagnostics + navigation over JSON-RPC/stdio.

**Why a separate binary:**
- LLVM adds ~30MB to the binary and ~200ms startup time. The LSP never needs LLVM.
- The LSP needs to be a long-running process — the compiler is fire-and-forget
- Different dependency profile: LSP needs JSON serialization + stdin/stdout protocol handling
- Clean separation: compiler changes automatically improve LSP (shared typer)

**Why not a dynamic library:** No benefit. The LSP is the only consumer. Static linking is simpler, faster, and eliminates deployment headaches.

### 3.2 Binary Setup

Add to `Cargo.toml`:

```toml
[[bin]]
name = "jadec-lsp"
path = "src/lsp/main.rs"

[dependencies]
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"
```

`serde` + `serde_json` are the only new dependencies. ~50KB compiled. The LSP protocol is JSON-RPC over stdio — serde handles serialization, the rest is hand-written protocol logic. No lsp-server framework crate needed (those add 100+ transitive deps for marginal benefit).

### 3.3 LSP Protocol Implementation

#### 3.3.1 JSON-RPC Transport — `src/lsp/transport.rs` (~120 LOC)

```rust
pub struct Transport {
    reader: BufReader<Stdin>,
    writer: BufWriter<Stdout>,
}

impl Transport {
    pub fn new() -> Self { ... }

    pub fn read_message(&mut self) -> Result<Message, String> {
        // Read "Content-Length: N\r\n\r\n" header
        let mut header = String::new();
        self.reader.read_line(&mut header)?;
        let len: usize = header.trim()
            .strip_prefix("Content-Length: ")?
            .parse()?;
        let mut blank = String::new();
        self.reader.read_line(&mut blank)?; // consume \r\n
        // Read exactly N bytes of JSON
        let mut body = vec![0u8; len];
        self.reader.read_exact(&mut body)?;
        let msg: Message = serde_json::from_slice(&body)?;
        Ok(msg)
    }

    pub fn send_message(&mut self, msg: &Message) -> Result<(), String> {
        let body = serde_json::to_string(msg)?;
        write!(self.writer, "Content-Length: {}\r\n\r\n{}", body.len(), body)?;
        self.writer.flush()?;
        Ok(())
    }
}
```

**Why hand-rolled transport:** The LSP content-length framing protocol is trivial — two `read_line` calls and one `read_exact`. A framework would pull in async runtime, tower, tokio — massive overkill for a stdio pipe. Keep it synchronous, single-threaded, zero-allocation on the hot path.

#### 3.3.3 Analysis Engine — `src/lsp/analysis.rs` (~300 LOC)

The core of the LSP. Runs the compiler pipeline without codegen and extracts semantic information.

```rust
pub struct AnalysisResult {
    pub diagnostics: Vec<LspDiagnostic>,
    pub symbols: Vec<DocumentSymbol>,
    pub definitions: HashMap<(u32, u32), Span>,  // (line, col) → definition span
    pub hover_info: HashMap<(u32, u32), String>,  // (line, col) → type info
}

pub fn analyze(uri: &str, source: &str) -> AnalysisResult {
    let mut result = AnalysisResult::default();

    // Stage 1: Lex
    let tokens = match Lexer::new(source).tokenize() {
        Ok(t) => t,
        Err(e) => {
            result.diagnostics.push(lex_error_to_diagnostic(e));
            return result;
        }
    };

    // Stage 2: Parse
    let prog = match Parser::new(tokens.clone()).parse_program() {
        Ok(p) => p,
        Err(e) => {
            result.diagnostics.push(parse_error_to_diagnostic(e));
            return result;
        }
    };

    // Stage 3: Extract document symbols from AST
    result.symbols = extract_symbols(&prog);

    // Stage 4: Type check
    let mut typer = Typer::new();
    let hir_prog = match typer.lower_program(&prog) {
        Ok(h) => h,
        Err(e) => {
            result.diagnostics.push(type_error_to_diagnostic(e));
            return result; // Partial results: symbols from AST still available
        }
    };

    // Stage 5: Build definition map from HIR
    result.definitions = build_def_map(&hir_prog, &typer);

    // Stage 6: Build hover map from HIR + typer
    result.hover_info = build_hover_map(&hir_prog, &typer);

    // Stage 7: HIR validation (non-fatal, add as warnings)
    let hir_errors = HirValidator::validate(&hir_prog);
    for e in hir_errors {
        result.diagnostics.push(hir_error_to_diagnostic(e));
    }

    // Stage 8: Ownership check (warnings)
    let mut verifier = OwnershipVerifier::new();
    let ownership_diags = verifier.verify(&hir_prog);
    for d in ownership_diags {
        result.diagnostics.push(ownership_diag_to_diagnostic(d));
    }

    result
}
```

**Critical design decision: No incremental analysis in v1.** The full pipeline (lex → parse → type check) runs on every keystroke (debounced to 300ms). For a 15K LOC codebase this takes <50ms. Incremental parsing is 5K+ LOC of complexity (Roslyn, rust-analyzer's green tree) — defer to 0.6.0 when/if latency becomes a problem. Keep it simple, keep it correct.

#### 3.3.4 LSP Server Main Loop — `src/lsp/main.rs` (~250 LOC)

```rust
fn main() {
    let mut transport = Transport::new();
    let mut documents: HashMap<String, String> = HashMap::new();  // uri → source text

    loop {
        let msg = match transport.read_message() {
            Ok(m) => m,
            Err(_) => break, // stdin closed
        };

        let method = match &msg.method {
            Some(m) => m.as_str(),
            None => continue, // Response, ignore
        };

        match method {
            "initialize" => handle_initialize(&mut transport, &msg),
            "initialized" => { /* no-op */ },
            "textDocument/didOpen" => {
                let params: DidOpenParams = serde_json::from_value(msg.params.unwrap()).unwrap();
                let uri = params.text_document.uri.clone();
                let text = params.text_document.text.clone();
                documents.insert(uri.clone(), text.clone());
                publish_diagnostics(&mut transport, &uri, &text);
            }
            "textDocument/didChange" => {
                let params: DidChangeParams = serde_json::from_value(msg.params.unwrap()).unwrap();
                let uri = params.text_document.uri.clone();
                let text = params.content_changes[0].text.clone();
                documents.insert(uri.clone(), text.clone());
                publish_diagnostics(&mut transport, &uri, &text);
            }
            "textDocument/didClose" => {
                let params: DidCloseParams = serde_json::from_value(msg.params.unwrap()).unwrap();
                documents.remove(&params.text_document.uri);
            }
            "textDocument/hover" => {
                handle_hover(&mut transport, &msg, &documents);
            }
            "textDocument/definition" => {
                handle_goto_definition(&mut transport, &msg, &documents);
            }
            "textDocument/documentSymbol" => {
                handle_document_symbols(&mut transport, &msg, &documents);
            }
            "textDocument/completion" => {
                handle_completion(&mut transport, &msg, &documents);
            }
            "shutdown" => {
                transport.send_message(&Message::result(msg.id, serde_json::Value::Null));
            }
            "exit" => break,
            _ => { /* ignore unknown methods */ }
        }
    }
}
```

#### 3.3.5 Feature Implementations

**Diagnostics (textDocument/publishDiagnostics):**

```rust
fn publish_diagnostics(transport: &mut Transport, uri: &str, source: &str) {
```

Fires on every `didOpen` and `didChange`. The analysis engine returns all diagnostics from all compiler stages (lex, parse, type check, ownership). The editor shows squiggly lines.

**Hover (textDocument/hover):**

```rust
fn handle_hover(transport: &mut Transport, msg: &Message, documents: &HashMap<String, String>) {
    let params: HoverParams = parse_params(msg);
    let uri = &params.text_document.uri;
    let source = match documents.get(uri) { Some(s) => s, None => return };
    let result = analyze(uri, source);
    let line = params.position.line;
    let col = params.position.character;

    // Find token at position
    let hover_text = if let Some(info) = result.hover_info.get(&(line, col)) {
        info.clone()
    } else {
        // Fallback: find nearest symbol
        find_token_type_at(source, &result, line, col)
    };

    transport.send_response(msg.id.clone(), json!({
        "contents": { "kind": "markdown", "value": hover_text }
    }));
}
```

The hover map is built during analysis. For each `Expr::Var`, `Expr::Call`, `Expr::Field`, `Expr::Method` in the HIR, record `(line, col) → type_string`. For variables: `"x: i64"`. For functions: `"*fib(n: i64) -> i64"`. For fields: `"Point.x: f64"`.

**Building the hover map from the Typer:**

The Typer already has all scope information (`scopes: Vec<HashMap<String, VarInfo>>`). During `lower_program`, we need a side-channel that records span → type mappings. Add a `pub(crate) span_types: Vec<(Span, Type)>` field to Typer. At each variable binding, function call, and field access, push the span and resolved type. The LSP then builds its `hover_info` map from this.

This is a **~20-line change** to the typer: add the field, and insert into it at the ~5 key sites (Bind, Var, Call, Method, Field).

**Go to Definition (textDocument/definition):**

```rust
fn handle_goto_definition(transport: &mut Transport, msg: &Message, docs: &HashMap<String, String>) {
    let params: DefinitionParams = parse_params(msg);
    let source = match docs.get(&params.text_document.uri) { Some(s) => s, None => return };
    let result = analyze(&params.text_document.uri, source);
    let pos = (params.position.line, params.position.character);

    let location = if let Some(def_span) = result.definitions.get(&pos) {
        Some(Location {
            uri: params.text_document.uri.clone(),
            range: span_to_range(def_span),
        })
    } else {
        None
    };

    transport.send_response(msg.id.clone(), serde_json::to_value(&location).unwrap());
}
```

The definition map is built from the HIR's `DefId` system. Every `Expr::Var(def_id, name)` carries its `DefId`. The typer knows where each `DefId` was defined (from `define_var` calls). Record `(use_span) → (def_span)` mappings during lowering. ~10-line change to typer.

**Document Symbols (textDocument/documentSymbol):**

Extracted from the AST, not the HIR. Walk `prog.decls` and emit:
- `Decl::Fn(f)` → `SymbolKind::Function`, name = `f.name`, range = `f.span`
- `Decl::Type(td)` → `SymbolKind::Struct`, children = fields + methods
- `Decl::Enum(ed)` → `SymbolKind::Enum`, children = variants
- `Decl::Extern(ef)` → `SymbolKind::Function` with "extern" detail

Pure AST walk, ~40 lines. No type checking needed.

**Completion (textDocument/completion):**

Basic completion from the typer's scope:
1. At cursor position, identify the current scope (function body, top-level, method)
2. Collect all names in scope: local variables, function names, type names, enum variants
3. After `.` → look up the type of the expression before the dot, return its fields/methods
4. Return as `CompletionItem[]` with kind (Variable, Function, Field, Method)

For v1, keyword completion + scope-based completion is sufficient. No fuzzy matching, no AI ranking. Just dump what's in scope. ~80 lines.

## 4. P2 — Compile-Time Evaluation

### 4.1 Design Philosophy

`comptime` is not a macro system. It's not a template language. It's the Jade compiler executing Jade code at compile time and splicing the results into the program. Same language, same semantics, same type system — just evaluated during compilation instead of at runtime.

**The Zig model, not the C++ model:** Zig's `comptime` is the gold standard. It uses the same evaluation engine as the runtime, operates on the same types, and produces the same results. C++'s `constexpr`/`consteval` is a crippled subset with different rules, different error messages, and different capabilities at each standard revision. We want the Zig model.

**What comptime enables:**
- Static assertions: `comptime { assert(size_of(Point) equals 16) }`
- Compile-time lookup tables: `comptime { TABLE is [sin(i as f64 * 0.01) for i from 0 to 628] }`
- Conditional compilation: `comptime { if PLATFORM equals "linux" { ... } }`
- Code generation: `comptime { for field in fields_of(MyStruct) { ... } }`
- Configuration: `comptime { MAX_THREADS is 8 }`

### 4.2 Syntax

```jade
# Block form — evaluated at compile time, results spliced into program
comptime
    TABLE_SIZE is 256
    SIN_TABLE is [sin(i as f64 / 256.0 * 2.0 * PI) for i from 0 to TABLE_SIZE]

# Inline form — single expression
MASK is comptime (1 << 16) - 1

# Static assertion
comptime
    assert(size_of(Point) equals 16, "Point must be 16 bytes")
```

### 4.3 Implementation Strategy: AST Interpreter

**Why an interpreter, not LLVM JIT:**
- LLVM JIT (ORC/MCJIT) adds ~200ms startup + 30MB memory overhead per compilation
- Comptime blocks are typically tiny (< 100 expressions) — interpreter overhead is negligible
- The interpreter operates on the AST/HIR, not machine code — it can introspect types, fields, sizes
- Zig does exactly this: their comptime evaluator is an AST interpreter, not a JIT
- An interpreter is ~500 LOC. A JIT integration is ~2000 LOC + massive complexity

**Why not a bytecode VM:** For the same reason. Comptime expressions are small. The overhead of compiling to bytecode and then interpreting bytecode exceeds the overhead of directly walking the AST. A tree-walking interpreter on a small expression tree is faster than a bytecode VM for < 10K operations. When/if comptime blocks grow to millions of operations (they won't), revisit.

#### 4.4.1 Value Representation

#### 4.4.2 Interpreter Core

```rust
pub struct ComptimeInterpreter {
    vars: HashMap<String, CTValue>,
    fns: HashMap<String, ast::Fn>,      // functions available at comptime
}

impl ComptimeInterpreter {
    pub fn new() -> Self { ... }

    pub fn eval_block(&mut self, stmts: &[ast::Stmt]) -> Result<CTValue, String> {
        let mut result = CTValue::Void;
        for stmt in stmts {
            result = self.eval_stmt(stmt)?;
        }
        Ok(result)
    }

    fn eval_stmt(&mut self, stmt: &ast::Stmt) -> Result<CTValue, String> {
        match stmt {
            Stmt::Bind(name, expr, _) => {
                let val = self.eval_expr(expr)?;
                self.vars.insert(name.clone(), val.clone());
                Ok(val)
            }
            Stmt::Expr(expr) => self.eval_expr(expr),
            Stmt::If(cond, then, elifs, els, _) => {
                if self.eval_expr(cond)?.as_bool()? {
                    self.eval_block(then)
                } else {
                    for (ec, eb) in elifs {
                        if self.eval_expr(ec)?.as_bool()? {
                            return self.eval_block(eb);
                        }
                    }
                    if let Some(eb) = els {
                        self.eval_block(eb)
                    } else {
                        Ok(CTValue::Void)
                    }
                }
            }
            Stmt::For(bind, start, end, step, body, _) => {
                let s = self.eval_expr(start)?.as_int()?;
                let e = self.eval_expr(end)?.as_int()?;
                let st = step.as_ref().map(|x| self.eval_expr(x)).transpose()?.map(|v| v.as_int()).transpose()?.unwrap_or(1);
                let mut i = s;
                while i < e {
                    self.vars.insert(bind.clone(), CTValue::Int(i));
                    self.eval_block(body)?;
                    i += st;
                }
                Ok(CTValue::Void)
            }
            _ => Err(format!("unsupported statement in comptime: {:?}", stmt)),
        }
    }

    fn eval_expr(&mut self, expr: &ast::Expr) -> Result<CTValue, String> {
        match expr {
            Expr::Int(n, _) => Ok(CTValue::Int(*n)),
            Expr::Float(f, _) => Ok(CTValue::Float(*f)),
            Expr::Str(s, _) => Ok(CTValue::Str(s.clone())),
            Expr::Bool(b, _) => Ok(CTValue::Bool(*b)),
            Expr::Ident(name, _) => {
                self.vars.get(name).cloned()
                    .ok_or_else(|| format!("undefined in comptime: {name}"))
            }
            Expr::BinOp(lhs, op, rhs, _) => {
                let l = self.eval_expr(lhs)?;
                let r = self.eval_expr(rhs)?;
                self.eval_binop(l, *op, r)
            }
            Expr::UnaryOp(op, e, _) => {
                let v = self.eval_expr(e)?;
                self.eval_unary(*op, v)
            }
            Expr::Call(name, args, _) => {
                let evaluated_args: Vec<CTValue> = args.iter()
                    .map(|a| self.eval_expr(a))
                    .collect::<Result<_, _>>()?;
                self.eval_call(name, &evaluated_args)
            }
            Expr::Array(elems, _) => {
                let vals: Vec<CTValue> = elems.iter()
                    .map(|e| self.eval_expr(e))
                    .collect::<Result<_, _>>()?;
                Ok(CTValue::Array(vals))
            }
            Expr::Tuple(elems, _) => {
                let vals: Vec<CTValue> = elems.iter()
                    .map(|e| self.eval_expr(e))
                    .collect::<Result<_, _>>()?;
                Ok(CTValue::Tuple(vals))
            }
            Expr::ListComp(body, bind, iter_start, iter_end, cond, _, _) => {
                let start = self.eval_expr(iter_start)?.as_int()?;
                let end = self.eval_expr(iter_end)?.as_int()?;
                let mut results = Vec::new();
                for i in start..end {
                    self.vars.insert(bind.clone(), CTValue::Int(i));
                    if let Some(c) = cond {
                        if !self.eval_expr(c)?.as_bool()? {
                            continue;
                        }
                    }
                    results.push(self.eval_expr(body)?);
                }
                Ok(CTValue::Array(results))
            }
            _ => Err(format!("unsupported expression in comptime: {:?}", expr)),
        }
    }

    fn eval_binop(&self, l: CTValue, op: BinOp, r: CTValue) -> Result<CTValue, String> {
        match (l, op, r) {
            (CTValue::Int(a), BinOp::Add, CTValue::Int(b)) => Ok(CTValue::Int(a + b)),
            (CTValue::Int(a), BinOp::Sub, CTValue::Int(b)) => Ok(CTValue::Int(a - b)),
            (CTValue::Int(a), BinOp::Mul, CTValue::Int(b)) => Ok(CTValue::Int(a * b)),
            (CTValue::Int(a), BinOp::Div, CTValue::Int(b)) => {
                if b == 0 { return Err("division by zero in comptime".into()); }
                Ok(CTValue::Int(a / b))
            }
            (CTValue::Int(a), BinOp::Mod, CTValue::Int(b)) => Ok(CTValue::Int(a % b)),
            (CTValue::Int(a), BinOp::Shl, CTValue::Int(b)) => Ok(CTValue::Int(a << b)),
            (CTValue::Int(a), BinOp::Shr, CTValue::Int(b)) => Ok(CTValue::Int(a >> b)),
            (CTValue::Int(a), BinOp::BitAnd, CTValue::Int(b)) => Ok(CTValue::Int(a & b)),
            (CTValue::Int(a), BinOp::BitOr, CTValue::Int(b)) => Ok(CTValue::Int(a | b)),
            (CTValue::Int(a), BinOp::BitXor, CTValue::Int(b)) => Ok(CTValue::Int(a ^ b)),
            (CTValue::Int(a), BinOp::Pow, CTValue::Int(b)) => Ok(CTValue::Int(a.pow(b as u32))),
            (CTValue::Int(a), BinOp::Eq, CTValue::Int(b)) => Ok(CTValue::Bool(a == b)),
            (CTValue::Int(a), BinOp::Neq, CTValue::Int(b)) => Ok(CTValue::Bool(a != b)),
            (CTValue::Int(a), BinOp::Lt, CTValue::Int(b)) => Ok(CTValue::Bool(a < b)),
            (CTValue::Int(a), BinOp::Gt, CTValue::Int(b)) => Ok(CTValue::Bool(a > b)),
            (CTValue::Int(a), BinOp::LtEq, CTValue::Int(b)) => Ok(CTValue::Bool(a <= b)),
            (CTValue::Int(a), BinOp::GtEq, CTValue::Int(b)) => Ok(CTValue::Bool(a >= b)),
            (CTValue::Float(a), BinOp::Add, CTValue::Float(b)) => Ok(CTValue::Float(a + b)),
            (CTValue::Float(a), BinOp::Sub, CTValue::Float(b)) => Ok(CTValue::Float(a - b)),
            (CTValue::Float(a), BinOp::Mul, CTValue::Float(b)) => Ok(CTValue::Float(a * b)),
            (CTValue::Float(a), BinOp::Div, CTValue::Float(b)) => Ok(CTValue::Float(a / b)),
            (CTValue::Bool(a), BinOp::And, CTValue::Bool(b)) => Ok(CTValue::Bool(a && b)),
            (CTValue::Bool(a), BinOp::Or, CTValue::Bool(b)) => Ok(CTValue::Bool(a || b)),
            (CTValue::Str(a), BinOp::Add, CTValue::Str(b)) => Ok(CTValue::Str(a + &b)),
            _ => Err("type mismatch in comptime binop".into()),
        }
    }

    fn eval_call(&mut self, name: &str, args: &[CTValue]) -> Result<CTValue, String> {
        match name {
            // Builtin comptime functions
            "size_of" => { /* requires type info from typer — see §4.5 */ }
            "align_of" => { /* requires type info from typer */ }
            "assert" => {
                if !args[0].as_bool()? {
                    let msg = args.get(1)
                        .map(|v| v.as_str().unwrap_or("assertion failed"))
                        .unwrap_or("comptime assertion failed");
                    return Err(format!("comptime assert: {msg}"));
                }
                Ok(CTValue::Void)
            }
            "sin" => Ok(CTValue::Float(args[0].as_float()?.sin())),
            "cos" => Ok(CTValue::Float(args[0].as_float()?.cos())),
            "sqrt" => Ok(CTValue::Float(args[0].as_float()?.sqrt())),
            "abs" => match &args[0] {
                CTValue::Int(n) => Ok(CTValue::Int(n.abs())),
                CTValue::Float(f) => Ok(CTValue::Float(f.abs())),
                _ => Err("abs requires numeric argument".into()),
            },
            "log" => {
                // comptime log → compile-time print (for debugging comptime code)
                eprintln!("comptime: {:?}", args[0]);
                Ok(CTValue::Void)
            }
            _ => {
                // Try user-defined comptime function
                if let Some(f) = self.fns.get(name).cloned() {
                    self.eval_user_fn(&f, args)
                } else {
                    Err(format!("unknown comptime function: {name}"))
                }
            }
        }
    }

    fn eval_user_fn(&mut self, f: &ast::Fn, args: &[CTValue]) -> Result<CTValue, String> {
        // Push scope, bind params, eval body, pop scope
        let prev_vars = self.vars.clone();
        for (param, arg) in f.params.iter().zip(args) {
            self.vars.insert(param.name.clone(), arg.clone());
        }
        let result = self.eval_block(&f.body.stmts);
        self.vars = prev_vars;
        result
    }
}
```

#### 4.4.3 Execution Budget

Comptime interpretation has a hard instruction budget to prevent infinite loops:

```rust
const COMPTIME_MAX_STEPS: u64 = 1_000_000;

// In eval_expr:
self.steps += 1;
if self.steps > COMPTIME_MAX_STEPS {
    return Err("comptime evaluation exceeded 1M steps — possible infinite loop".into());
}
```

1M steps is enough for any reasonable comptime block (lookup tables, assertions, config). If you're doing 1M operations at compile time, you're doing it wrong.

### 4.5 Integration into the Compilation Pipeline

Comptime blocks execute **between parsing and type checking**. The AST is the input; comptime results replace the comptime blocks with their evaluated values.

```
Lexer → Parser → AST → **Comptime Eval** → AST' → Typer → HIR → ...
```

**Why before the typer:**
- Comptime can define constants used in type definitions (`comptime { SIZE is 256 }` → `type Buf: data: Array of i8, SIZE`)
- Comptime can conditionally include/exclude declarations
- The typer sees a clean AST with all comptime blocks replaced by their results

```rust
fn expand_comptime(prog: &mut Program, ct: &mut ComptimeInterpreter) -> Result<(), String> {
    let mut new_decls = Vec::new();
    for decl in &prog.decls {
        match decl {
            Decl::Comptime(stmts, span) => {
                // Evaluate and extract any bindings as top-level constants
                for stmt in stmts {
                    ct.eval_stmt(stmt).map_err(|e| format!("line {}: {e}", span.line))?;
                }
                // Bindings from comptime become Decl::Fn (constants) or are discarded
            }
            Decl::Fn(f) => {
                // Register function as available for comptime calls
                ct.fns.insert(f.name.clone(), f.clone());
                new_decls.push(decl.clone());
            }
            other => new_decls.push(other.clone()),
        }
    }
    // Replace comptime expressions within function bodies
    for decl in &mut new_decls {
        if let Decl::Fn(f) = decl {
            expand_comptime_in_body(&mut f.body, ct)?;
        }
    }
    prog.decls = new_decls;
    Ok(())
}

fn expand_comptime_in_body(body: &mut Block, ct: &mut ComptimeInterpreter) -> Result<(), String> {
    for stmt in &mut body.stmts {
        expand_comptime_in_stmt(stmt, ct)?;
    }
    Ok(())
}

fn expand_comptime_in_stmt(stmt: &mut Stmt, ct: &mut ComptimeInterpreter) -> Result<(), String> {
    // Walk the statement tree, replacing Expr::Comptime with evaluated literals
    walk_exprs_mut(stmt, &mut |expr| {
        if let Expr::Comptime(stmts, _) = expr {
            let val = ct.eval_block(stmts)?;
            *expr = ctvalue_to_expr(val)?;
        }
        Ok(())
    })
}

fn ctvalue_to_expr(val: CTValue) -> Result<Expr, String> {
    match val {
        CTValue::Int(n) => Ok(Expr::Int(n, Span::synthetic())),
        CTValue::Float(f) => Ok(Expr::Float(f, Span::synthetic())),
        CTValue::Bool(b) => Ok(Expr::Bool(b, Span::synthetic())),
        CTValue::Str(s) => Ok(Expr::Str(s, Span::synthetic())),
        CTValue::Array(elems) => {
            let exprs: Vec<Expr> = elems.into_iter()
                .map(ctvalue_to_expr)
                .collect::<Result<_, _>>()?;
            Ok(Expr::Array(exprs, Span::synthetic()))
        }
        CTValue::Void => Ok(Expr::None(Span::synthetic())),
        _ => Err("comptime result cannot be converted to expression".into()),
    }
}
```

## 7. Performance Considerations

### 7.1 Package Manager

- **Resolution:** O(n) in number of dependencies. No SAT solving, no backtracking. Sub-millisecond for typical projects (<20 deps).
- **Caching:** Shared global cache. Second build of same project: zero network I/O, zero file copies. Just path lookup.
- **Lockfile:** Linear parse, deterministic output. No sorting overhead (pre-sorted on write).

### 7.2 Standard Library

- **Zero overhead:** All std lib functions are inlined by LLVM after monomorphization. `std.math.lerp(a, b, t)` compiles to 2 multiplies and 2 adds — same as hand-written.
- **No runtime:** No initialization code, no constructors, no global state. First call is the first instruction.
- **FFI cost:** `extern` functions have no wrapper overhead. LLVM sees through the declaration and generates direct calls to libc symbols.

### 7.3 LSP

- **Full reanalysis:** ~50ms for 15K LOC (lex + parse + type check). Acceptable for 300ms debounce. No incremental infrastructure needed until ~100K LOC.
- **Memory:** One HashMap per open document. ~1MB for a typical 20-file project. The typer allocates ~5MB for a full analysis pass and drops it after.
- **Startup:** ~10ms (no LLVM initialization). The LSP binary is ~2MB stripped (vs ~35MB for the full compiler with LLVM).

### 7.4 Comptime

- **Interpreter speed:** ~100M steps/second for arithmetic operations (direct pattern matching on AST nodes). A 1000-element lookup table generates in ~10μs.
- **Budget:** 1M step hard limit prevents runaway compilation. Typical comptime blocks use <1000 steps.
- **Memory:** CTValue is stack-allocated for primitives, heap-allocated for arrays/strings. No GC needed — comptime allocations are dropped when the interpreter finishes.

## 8. Risk Analysis

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| git not available on target system | Package manager fails | Low | Error message: "git is required for package management" |
| Network unavailable during first build | Cannot fetch deps | Medium | `jadec fetch` as explicit step; vendoring support (copy to local) |
| Extern function name collisions across std modules | Linker errors | Medium | Prefix all externs: `__jade_io_fopen`, or rely on libc symbol namespace |
| LSP latency on large projects (>50K LOC) | Sluggish editor | Low (Jade projects are small now) | Add incremental parsing in 0.6.0 if needed |
| Comptime infinite recursion | Compiler hangs | Low | Step budget (1M) + recursion depth limit (64, matching typer) |
| Comptime side effects (file I/O, network) | Security concern | N/A | Comptime has no FFI access — only pure computation |
| String handling in std.io | Memory leaks if StringFromRaw not properly wired | Medium | Comprehensive tests for every string-producing function |

## 9. Non-Goals for 0.5.0

- **Semver ranges:** Not supporting `^1.0.0` or `~1.2.x`. Exact versions only.
- **Private registries:** No auth, no tokens. Public git repos only. Private repos work if git has ssh configured.
- **Incremental LSP:** Full reanalysis per keystroke. No incremental parsing, no green trees.
- **Comptime type manipulation:** No creating types at compile time. No metaprogramming beyond value computation.
- **Async LSP:** Single-threaded, synchronous. No concurrent request handling.
- **Package publishing:** No `jadec publish` command. Users push to git.
- **Dependency resolution with ranges:** No PubGrub, no version negotiation.
- **std.net:** TCP/UDP sockets deferred to 0.6.0 (requires async I/O from Phase 0.4.0).
